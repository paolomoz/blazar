What is now called "Adobe Experience Manager Edge Delivery Services" has been through at least six distinct naming phases since its inception. The product originated as Project Helix in mid-2018, was renamed to Project Franklin in approximately mid-2022, briefly marketed as Next-Gen Composability at Adobe Summit 2023, and formally named Edge Delivery Services in October 2023. Infrastructure domains evolved from hlx.page/hlx.live to aem.page/aem.live. The old hlx.live domain was blocked on December 18, 2025.

The naming history has left a significant archaeological trail: the adobe/helix-home GitHub repository itself documents six names the project has carried. Legacy references persist across the live site, GitHub repositories, npm packages, developer tools, and community documentation. The rebrand on aem.live content is approximately 93% complete, but the developer ecosystem retains substantial legacy naming, most notably the window.hlx JavaScript namespace, .helix/ configuration directories, and dozens of helix-* GitHub repositories that remain un-renamed.

🏷 The Complete Name Roster

The official architecture document in adobe/helix-home contains this remarkable footnote:

"This project has seen many names, including Helix, Project Helix, Project Franklin, Next Generation Composability, Success Edge, and Edge Delivery Services in Adobe Experience Manager Sites as a Cloud Service with document-based editing." — adobe/helix-home, architecture.md

Note the final name: "Edge Delivery Services in Adobe Experience Manager Sites as a Cloud Service with document-based editing" — at 89 characters, this may be one of the longest product names in enterprise software. In practice, it is shortened to "Edge Delivery Services" or "EDS."

Naming Paradox

As of February 2026, the internal architecture team still uses "Helix" to refer to versions of the platform architecture (Helix 1, 2, 3, 4, 5). The FAQ explicitly states: "since the general availability [of EDS], the name Franklin is no longer used, so Helix is referred to as the architecture and Adobe Experience Manager as the product." This means "Helix" is simultaneously the oldest and most current internal term, while the product-facing name has changed five times around it.

📅 Full Chronological Timeline

Era 1: Project Helix Mid-2018 – Mid-2022 Internal Codename

Origin: Project Helix emerged from Adobe's AEM engineering team as a radical experiment in content delivery. Named after the double helix structure of DNA, it represented a fundamental rethinking of how content management systems should work: instead of a monolithic Java application, content would flow from Google Docs or Microsoft Word through a serverless pipeline to the edge.

Architecture: The project evolved through three major architecture versions during this era:

  • Jun 2018 — First GitHub repository created: adobe/helix-pipeline (2018-06-28). The earliest artifact of the project.
  • Jul 2018 — adobe/helix-cli created (2018-07-05). The command-line tool originally invoked as hlx.
  • Summer 2018 — Hackathon 1 held in Salt Lake City, UT. First team gathering for the project.
  • Sep 2018 — project-helix.io documentation site created (2018-09-26). The project's first public-facing web presence.
  • Oct 2018 — helix-shared and vscode-helix created. Developer tooling begins.
  • 2018–2019 — Helix 1: A bring-your-own Adobe I/O runtime framework approach enabling server-side customization.
  • Jan 2019 — Hackathon 4 in Berkeley, CA. The team is growing.
  • Mar 2019 — adobe/helix-home created (2019-03-14) — becomes the canonical project hub.
  • 2019–2020 — Helix 2: Pivoted to SaaS model with client-side customization. Abandoned server-side extensibility.
  • Jul 2022 — Hacker News frontpage appearance (81 points, 123 comments). Community reaction is confused — many commenters cannot understand what the product does.
"I will never trust Adobe on their development platforms."
"By the time we understand it, it will be cancelled." — Hacker News commenters, July 4, 2022

Infrastructure domains: *.hlx.page (preview) and *.hlx.live (published). The hlx abbreviation derives directly from "helix." The project website lived at www.project-helix.io.

Era 2: Project Franklin Mid-2022 – Oct 2023 Renamed Codename

The Rename: In approximately mid-2022, the project was renamed from "Helix" to "Franklin." The name honored Rosalind Franklin, the scientist whose X-ray crystallography work was critical to discovering the double helix structure of DNA — maintaining the DNA naming theme from the original "Helix" codename.

"They chose to rename the project after Rosalind Franklin. She was a scientist whose work and research formed a critical foundation for understanding the molecular structure of DNA — the double helix." — Rightpoint (formerly Hoodoo Digital), "What is Adobe's Project Franklin?", September 16, 2022

Architecture:

  • 2022 — Helix/Franklin 3: SaaS with "superior availability and performance." The version naming itself shows the transition — this version is called both "Helix 3" and "Franklin 3."
  • Sep 2022 — Rightpoint publishes the first third-party explainer referring to "Project Franklin." The product is described as "still in its early stages."
  • Nov 2022 — Hackathon 10 (first post-COVID, held remotely). Project is now internally "Franklin."
  • Dec 2020 → 2022 — adobe/aem-boilerplate repo (originally helix-project-boilerplate, created 2020-12-11) is renamed. The old GitHub URL still redirects.
  • Jan 2023 — Cognizant Netcentric publishes early analysis calling it "AEM Franklin" and "Next Generation Composability."
  • Feb 2023 — adobe/franklin-plugin-perflogger created. The "franklin-" prefix appears on new repos.
  • Mar 2023 — adobe/franklin-sidekick-library created (2023-03-16).

Developer artifacts renamed: lib-franklin.js became the core client library. GitHub repos began using "franklin-" prefix. The CLI tool was still hlx. The "Franklin Organizer" tool launched. Infrastructure domains remained hlx.page and hlx.live.

VIP Program: During this era, select AEM Sites customers could access the product through the "Franklin VIP program," as described on the /business/demo page: "AEM Franklin is currently made available to select AEM Sites customers."

Interlude: Next-Gen Composability March 2023 Marketing Name

Adobe Summit 2023 (March 21, 2023): At Summit, Adobe announced a "revolutionary new release of Adobe Experience Manager (AEM)" that "redefines the Content Management System category." The technology that had been Project Franklin was presented under the marketing umbrella of "Next-Gen Composability" or "Next Generation Composability."

"This product has variously been called 'Project Helix', 'Adobe Franklin', 'AEM's Next-Gen Composability Features' and now (somewhat officially) is AEM Edge Delivery Service." — Arbory Digital, "What is AEM Edge Delivery Services?"
  • Mar 21, 2023 — Adobe Summit 2023 keynote. AEM reimagined announcement. The blog post does not actually use the term "Edge Delivery Services" yet.
  • Mar 21, 2023 — Perficient publishes blog titled "New From Adobe: Next-Gen Composability" with URL containing "adobe-edge-delivery-services-aka-next-gen-composability".
  • Apr 2023 — Netcentric publishes "reimagining content management for marketing teams" using "Next-Gen Composability."
  • May 2023 — adobe/franklin-dashboard created. The Franklin name still used in tooling.
  • Aug 2023 — Bounteous publishes "Introducing Adobe Edge Delivery Services: A New Era in Content Authoring." The EDS name starts gaining traction before the official GA.
  • Aug 2023 — Hackathon 11 in New York City. Team gathering during the naming transition.
  • Sep 2023 — adaptTo() 2023 conference: Sessions use both "Edge Delivery Services" and "AEM Franklin" — the naming transition is actively in progress.

This period is the peak of naming confusion. Partner blogs from this period use all of the following in their titles: "Next-Gen Composability," "AEM Franklin," "Edge Delivery Services," "Project Helix," and combinations thereof. One Netcentric blog from August 2023 is literally titled "navigating support challenges with AEM Franklin" while the product is already being called EDS elsewhere.

Era 3: Edge Delivery Services October 2023 – Present Official Product Name

General Availability: In October 2023, at Adobe MAX, Adobe officially named the product "Edge Delivery Services" as part of Adobe Experience Manager. This marked the end of all codenames and the beginning of a unified product identity.

"In October 2023, Adobe officially introduced Edge Delivery Services and document-based authoring as part of Adobe Experience Manager, marking the transition from the Franklin and Helix project names to a fully integrated solution." — aem.live/docs/faq
  • Oct 2023 — Adobe MAX 2023. "Edge Delivery Services" becomes the official product name.
  • Oct 2023 — Architecture version formally becomes Franklin 4: dual-stack architecture (AWS + Cloudflare delivery).
  • Nov 2023 — adobe/aem-sidekick created (2023-11-10) — new repo under AEM naming. The old helix-sidekick-extension is archived.
  • Nov 2023 — Adobe Developers Live 2023 features "Edge Delivery and Document-based Authoring Deep Dive" session.
  • ~2023 — CLI renamed from @adobe/helix-cli to @adobe/aem-cli starting at version 15.0.0. Old npm package deprecated. Binary still provides hlx alias for backwards compatibility.
  • ~2023 — Core client library renamed from lib-franklin.js to aem.js.
  • Jan 2024 — Hackathon 12 in Palm Springs under the EDS banner.
  • Mar 2024 — Adobe Summit 2024: Multiple EDS sessions including "Accelerating CMS projects with Edge Delivery Services in AEM Sites" and "Beyond the Hype: Edge Delivery Services for Measurable Business Impact."

Developer tooling migration: The @adobe/helix-cli npm package stopped at version 14.26.0 and was deprecated. The replacement @adobe/aem-cli started at version 15.0.0. Both the aem and hlx commands work, ensuring backwards compatibility.

Era 4: The aem.live Domain Migration 2024 – Dec 2025 Infrastructure Rebrand

The final frontier of the rebrand: While the product name changed in October 2023, the infrastructure domains took much longer. The new aem.live / aem.page architecture (internally called Helix 5) was nearly a year in development and over six months in production before the migration deadline.

"Edge Delivery Services started with Project Helix over five years ago, and the hlx abbreviation has been embedded throughout the platform ever since — in domain names, configuration keys, runtime objects and developer tools." — aem.live/blog/hlx-is-now-aem-live
  • 2024 — Helix 5 architecture deployed. Refactored Fastly CDN stack (single service instead of two stacked). New aem.page and aem.live domains activated.
  • 2024 — New projects default to aem.live architecture. Existing projects given migration path.
  • 2024 — project-helix.io domain now 301 redirects to www.aem.live.
  • Nov 2024 — Adobe Developers Live 2024: "aem.live at Adobe Developers Live" session showcased.
  • Late 2025 — Fewer than 2% of daily traffic still flows through hlx.live. Last dozen sites in active migration.
  • Dec 18, 2025 — hlx.live domain blocked. Franklin 4 end of service. All traffic must flow through aem.live.

What changed technically:

  • Published content: *.hlx.live → *.aem.live
  • Preview: *.hlx.page → *.aem.page
  • Identifiers shift from GitHub owner/repo model to AEM org/site model
  • Inner CDN services use new domains: rum.aem-fastly.page, media.aem-fastly.page, config.aem-fastly.page, etc.
  • Faster preview caching on aem.page vs. hlx.page
  • New capabilities enabled: centralized configurations, repoless sites, more secure authentication
"With aem.live and repoless sites, we were able to meet that demand and help our business grow." — The Channel Company, web development director, quoted on aem.live
Present Day: February 2026 Ongoing Current

As of February 2026, the product is marketed as "Edge Delivery Services" (the "AEM" prefix was dropped per an April 2, 2025 naming update from Adobe). The architecture is at Helix 5. Hackathon 19 is scheduled for February 2-6, 2026, in Palm Springs. The team continues to use "Helix" internally for architecture versioning while the public-facing name is stable for the first time in years.

  • Apr 2, 2025 — Adobe drops "AEM" prefix. The product is now just "Edge Delivery Services," though it remains part of the AEM portfolio.
  • Feb 2026 — Hackathon 19 in Palm Springs. The project's 19th team gathering, still running strong.
  • Feb 2026 — status.hlx.live still exists but redirects (302) to www.aemstatus.net.

Redirect Gap: hlx.live Returns 403 Not 301

The old hlx.live and hlx.page domains return a 403 Forbidden response with x-error: blocked, not a 301 redirect. This means any external links, bookmarks, or search engine indexes pointing to old hlx.live URLs will fail silently rather than forwarding users to the correct location. From an SEO perspective, a 403 is worse than a 301 — it signals to search engines that the content is gone rather than moved, causing link equity to be lost.

🔎 SEO Consequences of the Name Changes

Critical: hlx.live Returns 403, Not 301

The single biggest SEO risk from the domain migration: old hlx.live and hlx.page URLs return a 403 Forbidden instead of a 301 Permanent Redirect to the equivalent aem.live / aem.page URL. This means:

  • All inbound links from external sites pointing to hlx.live URLs lose their link equity
  • Search engine indexes of old URLs will show dead results until deindexed
  • Developer bookmarks and documentation links from the Helix/Franklin era are broken
  • GitHub README files in thousands of forked repos that reference hlx.live URLs are now dead

Keyword Fragmentation

The multiple names have created a fragmented search landscape. People searching for information about this product use many different terms:

  • "edge delivery services" — current official name
  • "aem edge delivery services" — previous full official name
  • "adobe franklin" / "aem franklin" — still used in many third-party articles
  • "project helix adobe" — still searched by developers familiar with early days
  • "adobe EDS" — common abbreviation
  • "next-gen composability" — some 2023 content still uses this

This fragmentation dilutes SEO authority. The aem.live FAQ page itself leads its meta description with the deprecated names "Franklin and Helix" — which, while helpful for discoverability, reinforces the old branding in search results.

Third-Party Content Lag

Partner and community blogs from 2022-2023 are a permanent part of the search landscape. Blog titles like:

  • "What is AEM Edge Delivery Services/Helix/AEM Franklin/Next Gen Composability?" (Arbory Digital)
  • "AEM Franklin / Helix / Composability: Pros and Cons" (Adobe Experience League Community)
  • "Navigating support challenges with AEM Franklin" (Netcentric, Aug 2023)
  • "Adobe's AEM Edge Delivery (previously AEM Franklin)" (Netcentric, Jan 2023)

These articles remain indexed and continue to introduce new users to old naming. Adobe cannot control third-party content, meaning the naming confusion will persist in search results for years.

Documentation URL Stability

Some Adobe Experience League documentation URLs still contain "franklin" in their paths. For example:

👥 Community & Developer Impact

The Discoverability Problem

"Part of the reason that finding good information about AEM Edge Delivery Services is difficult is that it's a product that's had so many different names over its short lifetime." — Arbory Digital

This quote encapsulates the core developer experience problem. When searching for help, developers must try multiple name variants. Stack Overflow answers from the Helix era may use different APIs. Blog tutorials from the Franklin era reference lib-franklin.js instead of aem.js. The constant renaming has fractured the knowledge base.

Hacker News Reception (July 2022)

When Project Helix reached Hacker News (81 points, 123 comments), the reaction was largely confusion:

"I will never trust Adobe on their development platforms." — HN commenter
"By the time we understand it, it will be cancelled." — HN commenter, referencing Adobe's history of discontinuing projects

Multiple commenters reported that the project website had broken links, unfinished pages, and failed to clearly explain what the product does. The general impression was of a project not ready for public scrutiny.

Developer Documentation Gaps

"There is not much information available about this ambitious project from Adobe, with very little documentation on the implementation and no community help content either." — Adobe Experience League Community

During the Franklin era, developers reported difficulty finding accurate information. The renaming compounded this: documentation written for "Helix" used different terminology than documentation written for "Franklin," and community forum answers from one era might not apply to the next.

💡 Strategic Analysis

Why So Many Names?

The naming evolution reflects a common pattern in enterprise software: an internal engineering project gradually becoming a commercial product.

NameTrigger for ChangeAudience
Project Helix Engineering team needed an internal codename Internal Adobe engineers
Project Franklin Helix 3 architecture was a significant enough change to warrant a new identity; also possibly to distance from the original's mixed reception Internal + early partners
Next-Gen Composability Adobe Summit marketing needed a customer-facing name that connected to broader AEM messaging Marketing / enterprise buyers
Success Edge Unclear — appears briefly in internal documentation. Possibly a discarded marketing candidate. Unknown
AEM Edge Delivery Services Product reached GA; needed a permanent, descriptive, AEM-aligned name Everyone — commercial launch name
Edge Delivery Services Adobe simplified branding by dropping redundant "AEM" prefix (the product is already under AEM) Everyone — current preferred name

Is the Current Naming Clear and Consistent?

The good:

  • "Edge Delivery Services" is descriptive — it tells you what it does (delivers at the edge)
  • The domain aem.live is short, memorable, and brand-aligned
  • The FAQ page explicitly addresses naming history, showing transparency
  • The CLI now supports both aem and hlx, enabling gradual transition

The bad:

  • The full official name — "Edge Delivery Services in Adobe Experience Manager Sites as a Cloud Service with document-based editing" — is 89 characters
  • "EDS" abbreviation competes with many other acronyms
  • window.hlx, .helix/, and .hlxignore create cognitive dissonance for new developers
  • 30+ GitHub repos still named "helix-*" signal a project that has not committed to its own rebrand
  • Some documentation pages simultaneously use "Edge Delivery Services", "Helix 5", "AEM", and "franklin" in different contexts

The structural: The decision to keep "Helix" as the internal architecture designator while using "Edge Delivery Services" as the product name is pragmatic but creates a permanent two-name system. Every developer who reads the architecture docs learns "Helix" as the primary concept, then must translate to "EDS" in all external communication.

Architecture Versions vs. Product Naming

The architecture versioning itself shows the naming confusion:

VersionInternal NameEraKey Change
1Helix 12018-2019Adobe I/O Runtime, server-side customization
2Helix 22019-2021SaaS, client-side customization only
3Helix/Franklin 32021-2022Improved availability and performance
4Franklin 42022-2025Dual-stack (AWS + Cloudflare), production grade
5Helix 52024-presentRefactored CDN, aem.live domains, repoless sites

Note how the naming oscillates: Helix → Helix → Helix/Franklin → Franklin → back to Helix. Version 5 reverted to "Helix" because "Franklin" was officially retired as a public name, but the internal architecture needed a label. The result: the most modern version carries the oldest name.

Recommendations

  1. Rename key GitHub repos: helix-website → aem-website, helix-cli → aem-cli, helix-home → eds-home or archive it. GitHub handles redirects automatically.
  2. Update the 5 live pages with updateable legacy references (especially /business/demo which still says "AEM Franklin" and /tools/organizer/ titled "Franklin Organizer").
  3. Add 301 redirects on hlx.live and hlx.page instead of returning 403. This preserves SEO link equity and user experience.
  4. Create canonical "what is EDS" page that ranks for all historical name variants and redirects searchers to the correct, current information.
  5. Plan window.hlx deprecation path: Introduce window.aem alias, maintain both for 2 years, then deprecate hlx.
  6. Standardize partner communication: Provide partners with a naming style guide to prevent future fragmentation in third-party content.
