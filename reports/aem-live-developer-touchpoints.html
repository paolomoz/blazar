Key Finding

The aem.live brand voice is strongest in the newest touchpoints (Sidekick extension, AGENTS.md) and weakest in the infrastructure layer (CLI repo naming, npm package metadata, legacy CONTRIBUTING.md files). The core tension: the product was renamed from "Project Helix" to "Franklin" to "AEM Edge Delivery Services," but the GitHub infrastructure still carries heavy Helix DNA. Users installing @adobe/aem-cli are actually cloning adobe/helix-cli, the startup banner says "AEM simulator," environment variables once required HLX_ prefix (now AEM_), and the ignore file is still called .hlxignore.

Naming Debt Summary

Of 12 key naming touchpoints examined, only 4 (33%) exclusively use the AEM brand. The remaining 8 expose "helix" or "hlx" to developers in daily workflows. The most impactful: window.hlx (every page load), .hlxignore (every project), admin.hlx.page (every Sidekick interaction), and @adobe/helix-rum-enhancer (every RUM-enabled page).

README.md, Documentation section

Issues: Contains a typo ("recommand" instead of "recommend"). Links to the tutorial, anatomy of a project, web performance, and blocks documentation -- all on-brand references. The smiley face at the end (:)) shows Controlled Playfulness.

Contributing Guidelines

"# Contributing to Project Helix

This project (like almost all of Project Helix) is an Open Development project..."

CONTRIBUTING.md, line 1

Critical brand violation: The CONTRIBUTING.md still says "Project Helix" in the title and body. A developer contributing to what they believe is an AEM project is immediately told they are contributing to "Project Helix." This creates significant brand confusion.

Code Comment Style

The core library (scripts/aem.js) uses clean JSDoc patterns:

/**
 * Sanitizes a string for use as class name.
 * @param {string} name The unsanitized string
 * @returns {string} The class name
 */

Comments are terse and functional -- no humor, no brand voice, just engineering documentation. Error handling uses silent patterns:

} catch (error) {
    // something went awry
}

Block code (cards.js, header.js) uses inline comments sparingly: /* change to ul, li */. This aligns with Engineering Pragmatism.

File/Folder Conventions

  • blocks/{blockname}/{blockname}.js -- consistent pattern
  • scripts/aem.js -- core lib uses current brand name
  • .hlxignore -- LEGACY still uses old prefix
  • window.hlx in runtime -- LEGACY
  • AGENTS.md + CLAUDE.md -- forward-thinking AI agent support

The boilerplate also includes helix-rum-enhancer as a runtime dependency loaded from ot.aem.live.

adobe/aem-block-collection

BC
Extended block library for AEM projects
JavaScript | Apache-2.0 | Created 2022-07-15 | Last updated 2026-02-03

README: Carbon Copy Problem

The block collection README is nearly identical to the boilerplate README. Same placeholder title, same documentation links, same smiley face, same typo ("recommand"). This is not just template inheritance -- it is copy-paste without adaptation.

"Before using the aem-block-collection, we recommand you to go through the documentation..."

README.md -- identical structure to boilerplate README

A block collection should have its own identity: what blocks are included, how to use them, visual previews. Instead, a developer landing on this repo sees a generic template README.

Code Quality

Block implementations (accordion, carousel, embed, form, modal, quote, search, table, tabs, video) are well-structured:

  • Consistent naming: blocks/{name}/{name}.js + {name}.css
  • No unnecessary dependencies
  • Accessibility considered (ARIA attributes in carousel, keyboard navigation)
  • Zero code comments in most blocks -- code speaks for itself

The code itself embodies Engineering Pragmatism perfectly: functional, minimal, zero abstraction layers.

Missing Brand Elements

  • No block gallery/documentation -- developer must read code to understand each block
  • No visual examples or screenshots
  • CONTRIBUTING.md references "Project Helix" (same as boilerplate)
  • GitHub topics still include helix and helix-example
  • Description is just "AEM Block Collection" -- misses opportunity to convey brand value

Error Messages

Error messages are helpful and actionable -- strong Engineering Pragmatism:

  • "aem up needs local git repository." -- direct
  • "git submodules are not supported." -- clear constraint
  • "No git remote found. Make sure you have a remote 'origin' configured." -- diagnostic
  • "URL exceeds the 63 character limit for DNS labels. Please use a shorter branch name." -- explanation + action
  • "Unable to start AEM: {message}" -- wrapped with context

The troubleshooting section for TLS certificate issues is exceptionally well-written: explains the "why" (enterprise proxy), provides two options (IT-provided cert vs browser extraction), and includes step-by-step instructions for both Mac and Windows.

Command/Flag Naming

Command naming is clean and follows conventions:

  • aem up -- short, memorable, verb-first
  • aem import -- clear purpose
  • Flags use kebab-case: --log-level, --tls-cert, --forward-browser-logs
  • Sensible defaults: port 3000, livereload on, auto-open browser
  • Env var support: AEM_PORT, AEM_OPEN, etc.

Flag groups are well-organized: "Server options" and "AEM Options" -- clear conceptual separation.

Issue Templates

Three templates: bug report, feature request, and discussion. The bug report asks for:

"A clear and concise description of what the bug is... Steps to reproduce... Expected behavior... Version: run: $ aem --version"

.github/ISSUE_TEMPLATE/bug_report.md

Functional but entirely generic GitHub templates with no brand personality. Compare this to the website's confident, opinionated tone -- these templates could belong to any repo.

UI Text and Error Messages: The Best in the Ecosystem

The Sidekick's English locale file (messages.json) reveals the most carefully crafted brand voice of any developer touchpoint. Analysis of 200+ message strings:

Success Messages -- Confident and Clear

  • "Preview successfully updated, opening Preview..." -- result + next action
  • "Publication successful, opening $1..." -- confirms + redirects
  • "Page successfully deleted, reloading..." -- confirms + continues
  • "Project added to Sidekick" -- concise, no excess

Error Messages -- Helpful, Not Cryptic

  • "Apologies, we seem to be having problems at the moment. Please try again later." -- the only place in the entire ecosystem that says "apologies"
  • "Too many requests: Your project is being throttled by $1. Please wait and try again later." -- explains the cause
  • "This is a Microsoft Word document, please convert it to Google Docs first." -- specific and actionable
  • "File must be previewed first" -- tells you the prerequisite
  • "Access denied. Try signing in with a different user or ask your administrator for sufficient permissions." -- two alternative actions

Confirmation Prompts -- Respectful of Consequences

  • "This page no longer has a source document, deleting it cannot be undone! Are you sure you want to delete it?" -- warns about irreversibility
  • "Caution: unexpected side effects!" -- rare emotional language

Technical Error Messages -- Highly Specific

  • "Unable to preview $1: MP4 is longer than 2 minutes"
  • "Unable to preview $1: MP4 bit rate is higher than 300KB/s"
  • "Unable to preview $1: illegal scripting detected in SVG"
  • "Unable to preview $1: SVG root item missing"

These technical messages are the strongest expression of Performance Moralism in any developer touchpoint -- they communicate exact limits, not vague failures.

Heatmap Interpretation

Engineering Pragmatism is the one trait that survives consistently across all touchpoints (average 4.3/5). The codebase is the brand promise: no build steps, no frameworks, minimal abstraction.

Declarative Confidence drops sharply in repo-level artifacts (READMEs, npm metadata). The website says things like "Your website. But way faster." while the npm description says "AEM CLI".

Controlled Playfulness is the weakest trait overall (average 2.0/5). The website has moments of wit; the developer tools are relentlessly serious, with the exception of AGENTS.md's closing joke about frustrated humans.

The AGENTS.md is the gold standard -- it is the only touchpoint that matches the website across all 6 traits.

Critical Brand Inconsistencies

Specific issues that undermine brand coherence between the website and developer tools.

1. The window.hlx Runtime Namespace

Every AEM Edge Delivery Services page in production creates a window.hlx object. This is in the core aem.js library that every project inherits. It contains RUM data, configuration, and runtime state.

window.hlx = window.hlx || {};
window.hlx.RUM_MASK_URL = 'full';
window.hlx.RUM_MANUAL_ENHANCE = true;
window.hlx.codeBasePath = '';

Any developer debugging an AEM site will encounter window.hlx in the console. Combined with .hlxignore files and *.hlx.page preview URLs, a developer unfamiliar with the history would reasonably conclude the product is called "HLX," not "AEM."

Impact: Every developer, every day, every project.

2. CONTRIBUTING.md Brand Mismatch

Both the boilerplate and CLI repos have CONTRIBUTING.md files that reference "Project Helix." Since the boilerplate is a template repo, every new AEM project inherits a contribution guide for a product called "Helix."

The boilerplate says: "Contributing to Project Helix -- This project (like almost all of Project Helix) is an Open Development project."

The CLI says: "Contributing to Helix-CLI"

The Sidekick says: "Contributing to AEM Sidekick" -- the only correct one.

Impact: Every contributor to any EDS project.

3. npm Package Discoverability Gap

The @adobe/aem-cli package has:

  • No keywords (null)
  • A two-word description ("AEM CLI")
  • Homepage pointing to helix-cli
  • No mention of "Edge Delivery Services," "development server," or "local preview"

A developer searching npm for "edge delivery" or "aem development server" would not find this package. Compare to the website's SEO-rich descriptions.

Impact: Developer acquisition funnel.

4. Three Naming Eras Coexist

Active, non-archived repos in the Adobe org that use each name:

  • "helix-" prefix: helix-cli, helix-website, helix-importer-ui, helix-html-pipeline, helix-rum-enhancer, helix-rum-js, helix-rum-collector, helix-shared, helix-deploy, and 20+ more
  • "aem-" prefix: aem-boilerplate, aem-block-collection, aem-sidekick
  • "franklin-" prefix: franklin-sidekick-library

The "helix-" repos outnumber "aem-" repos roughly 8:1 in the active EDS ecosystem. This is an infrastructure artifact -- internal services were never renamed -- but it means any developer exploring the org encounters "Helix" far more than "AEM."

5. GitHub Topics Tag Legacy

GitHub topics on AEM repos still include helix, helix3, helix4, helix5, helix-example. These are searchable and visible in repo cards. The boilerplate -- the first repo most developers encounter -- carries 5 helix-era topic tags and zero "aem" or "edge-delivery" topics.

6. What Works Well: The Sidekick as Brand Exemplar

The Sidekick extension (created Nov 2023, newest of the repos) shows what post-rebrand developer artifacts look like when done right:

  • Repo named aem-sidekick (not helix-sidekick)
  • CONTRIBUTING.md says "Contributing to AEM Sidekick"
  • Extension title is "AEM Sidekick" everywhere
  • 200+ carefully written user-facing strings
  • 10-language localization
  • Technical error messages include specific file size limits and format requirements

The pattern is clear: newer artifacts are on-brand; older ones carry naming debt.

Overall Verdict

B+
Brand Consistency: Strong Code, Weak Metadata
The code itself is the best brand ambassador -- zero dependencies, no build steps, and transparent architecture deliver on every promise. But the surrounding metadata (repo names, package descriptions, CONTRIBUTING files, GitHub topics) actively undermines the brand narrative.

Strengths

  • Code quality matches brand promise of simplicity
  • Error messages in Sidekick are best-in-class
  • AGENTS.md is a masterclass in brand voice for AI assistants
  • CLI startup experience (ASCII banner) is polished
  • Three-phase loading pattern in boilerplate proves performance claims
  • Env vars migrated from HLX_ to AEM_

Weaknesses

  • window.hlx namespace in every page
  • CONTRIBUTING.md references "Project Helix" in template repo
  • npm metadata is nearly empty
  • CLI repo still named helix-cli
  • GitHub topics carry legacy naming
  • Block collection README is a generic copy-paste
  • admin.hlx.page in production
