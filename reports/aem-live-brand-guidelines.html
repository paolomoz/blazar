1. Brand Positioning

AEM Edge Delivery Services positions itself around four pillars. Every marketing message, testimonial, and documentation page reinforces these themes.

Performance-First CMS Primary Pillar

Speed dominates the brand. Lighthouse 100 appears in every customer testimonial. Performance is treated not as a feature but as a moral standard — the brand literally calls it "keeping it 100."

Familiar Tools, Not Replacements Differentiator

Authors use Word/Google Docs. Developers use GitHub. CDN is BYO. The platform integrates rather than replaces — "use the technology you already have."

Minimalism Philosophy Identity

Anti-complexity, anti-bloat. Explicitly rejects categories: not an SSG, not a headless CMS. "Everything modern web development needs, and not a single bit more."

Enterprise-Grade Simplicity Paradox

Balances enterprise credibility (PGA Tour, global CDN) with startup-like simplicity (10-20 minute setup, CLI-first, Discord community).

6. Documentation Patterns

Seven distinct content types identified. The site leans toward explanatory text and visual diagrams over code-heavy references.

1. Getting-Started Tutorial

Sequential 5-phase walkthrough. Copy-paste CLI commands with placeholders (<owner>, <repo>). Celebratory ending: "That's it, you made it!"

Example: /developer/tutorial

2. Guided References

Hybrid style: narrative flow like a tutorial but structured for lookups. Progressively explains concepts, then lists specific conventions.

Example: /developer/anatomy-of-a-project

3. Performance Guide

Prescriptive how-to with concrete targets (LCP under 1560ms, payload under 100kb). Problem-explanation-solution structure. No code — entirely conceptual.

Example: /developer/keeping-it-100

4. Catalog Pages

Index of blocks with classification labels (Default Content, Block, Autoblock, Deprecated). Two tiers: Boilerplate vs Collection.

Example: /developer/block-collection

5. Interactive Tool Views

Block detail pages redirect to Sidekick Library app. Not readable documentation — interactive preview tools. Invisible to search engines and LLMs.

Example: /developer/block-collection/columns

6. Author Guides

Task-oriented, minimal jargon, empathetic tone. Heavy on screenshots, light on code. Progressive disclosure pattern.

Example: /docs/authoring

7. Bridge Documents

Connect authoring and developer perspectives. Show how Word/Docs tables become HTML div structures with CSS classes.

Example: /docs/exploring-blocks

Media Bus Architecture (Strength)

Content-Addressable Storage

  • Every image stored once with a unique hash (e.g., media_1efffe0288d261...)
  • 4 variants auto-generated: 750px mobile + 2000px desktop, each in WebP + original format
  • <picture> elements with <source> for responsive delivery
  • Lazy loading by default, priority for LCP hero images
  • Same-origin delivery — no third-party CDN connection overhead
  • Dynamic manipulation via URL params: ?width=750&format=webply&optimize=medium

Irony of the Self-Serve Documentation

The /docs/authoring page explicitly tells authors: "It is a good idea to set an alternative text for all images" — while having zero alt text on its own images.
